这份手册讲什么

如果你经常听到大模型、RAG、Agent、MCP、AI Coding 这些词，但还没形成完整理解，这份手册会帮你用通俗语言把它们串起来：知道它们是什么、能解决什么问题、适合什么场景，以及落地时要注意什么。

01

AI 基础认知

  • 先分清 AI、大模型和生成式 AI，不再把它们混成一个概念。
  • 理解 AI 适合做什么，也知道哪些事情不能完全交给它。
  • 建立非技术同学也能使用的判断框架。
02

AI 热词速查

  • 把常见热词放在一处，遇到不懂的词可以快速查。
  • 每个词都用一句话先讲明白，再补充场景和误区。
  • 适合开会、看文章、研究工具时随手对照。
03

大模型 LLM

  • 用不写公式的方式理解 Token、上下文窗口和生成机制。
  • 知道为什么大模型会“看起来很懂”，也会一本正经地出错。
  • 学会判断一个模型能力强不强，不能只看演示效果。
04

模型与产品生态

  • 把 ChatGPT、Claude、Gemini、DeepSeek、Qwen、Llama 放到同一张图里看。
  • 分清“模型家族”和“面向用户的产品”不是一回事。
  • 选工具时先看场景，而不是只追最新名字。
05

Prompt 工程

  • 把“帮我写一下”变成 AI 真能执行的任务说明。
  • 学会给角色、目标、限制、格式和例子。
  • 减少反复改来改去，让输出更稳定。
06

上下文管理

  • 理解 AI 为什么有时“不记得”或“没看见”。
  • 学会给它足够但不过量的资料、背景和历史信息。
  • 避免把一堆无关内容塞进去，反而让结果变差。
07

RAG

  • 理解“让 AI 先查资料再回答”到底是怎么回事。
  • 知道它为什么常用于企业知识库和客服问答。
  • 看懂检索、引用和减少幻觉之间的关系。
08

知识库

  • 知道知识库不是把文档堆到一起就完事。
  • 理解资料质量、权限、更新和切分为什么重要。
  • 能判断一个知识库问答项目为什么好用或不好用。
09

Agent

  • 理解 AI 如何从“回答问题”走向“帮你做事”。
  • 看懂目标、计划、记忆和反馈在 Agent 中的作用。
  • 知道什么时候必须让人来确认，不能全自动。
10

Agent 工具生态

  • 把 Claude Code、Codex、OpenCode、OpenClaw、Hermes 等放在一起比较。
  • 知道不同 Agent 工具偏向代码、桌面、本地自动化还是 DevOps。
  • 生态变化很快，重要工具需要回到官方来源核验。
11

工具调用

  • 理解 AI 怎么查询订单、读文件、调用接口或运行脚本。
  • 知道工具调用为什么是 Agent 落地的关键能力。
  • 看清权限边界，避免 AI 做了不该做的事。
12

MCP

  • 理解 MCP 为什么被称为 AI 应用连接外部系统的标准方式。
  • 看懂 Tools、Resources、Prompts 分别解决什么问题。
  • 知道它和普通插件、API 调用有什么关系。
13

Skill

  • 理解 Skill 为什么像“给 AI 的任务方法说明书”。
  • 知道如何把团队经验沉淀成可复用流程。
  • 让 AI 做固定任务时更稳定，不用每次重新教一遍。
14

Agent Harness

  • 理解 harness 不是某个模型，而是让 Agent 稳定运行的支架。
  • 看懂状态、权限、评估、恢复和监控为什么重要。
  • 区分“harness 方法”和 Harness 这家公司/平台。
15

AI Coding 工作流

  • 理解 AI Coding 不是“自动写代码”，而是新的协作方式。
  • 学会描述需求、提供代码上下文、审查 AI 改动。
  • 看懂从需求到代码修改再到验证的完整流程。
16

Agentic Engineering

  • 理解为什么 AI 能做事以后，工程设计比 Prompt 更重要。
  • 看懂评估、日志、权限、回滚和人工确认的必要性。
  • 知道如何设计一个可控的人机协作系统。
17

工程要求

  • 知道一个 AI 项目为什么不能只看模型效果。
  • 理解数据、权限、成本、延迟和准确率都会影响落地。
  • 上线前要有评估、兜底和责任边界。

不需要从头背术语。更好的方式是先建立整体地图，再按自己关心的场景深入：想理解 AI 产品，就先看基础、RAG、Agent；想用 AI 做开发，就重点看上下文、AI Coding 和 Agentic Engineering。

    先快速看一遍 AI、生成式 AI、大语言模型、Prompt、RAG、Agent、AI Coding、Agentic Engineering，建立最基本的知识地图。

  2. 建立概念边界

    再补齐 Token、上下文窗口、幻觉、Embedding、向量数据库、Tool Use、Human-in-the-loop 和 Evaluation，知道哪些词容易混淆。

  3. 形成可讨论框架

    带着自己的业务或学习场景读核心模块，尝试判断哪些问题适合 RAG，哪些适合 Workflow，哪些才需要 Agent。

  4. 持续维护知识库

    以后遇到新词，不急着收藏一堆文章，先按“是什么、解决什么问题、适合什么场景、有什么限制”补进这份手册。

01 / Foundation

AI 基础认知

先建立正确预期：AI 不是单个工具，而是一类可以嵌入工作流的理解、生成和辅助执行能力。

是什么

AI 是一类让机器承担“理解、生成、判断、规划、执行”任务的技术集合。

  • 传统 AI 更偏识别、分类、预测，例如推荐系统、风控模型和图像识别。
  • 生成式 AI 更偏创造和表达，例如生成文本、图片、音频、视频和代码。
  • 大语言模型是当前最常见的能力底座，很多 AI 产品其实是在大模型上接入数据、工具和业务流程。
能力
  • 信息处理：总结长文、提取重点、对比材料、生成表格。
  • 表达生成：写初稿、改写语气、生成标题、整理汇报结构。
  • 辅助判断：提出备选方案、拆解问题、识别风险，但最终判断仍应由人负责。
  • 任务执行：接入工具后，可以搜索、读文件、调用接口、修改代码或触发流程。
场景
  • 个人学习：解释概念、生成学习路线、整理笔记、模拟问答。
  • 办公提效：会议纪要、周报、方案初稿、资料归纳。
  • 业务协作：用户反馈归类、竞品分析、客服知识库、运营活动文案。
  • 研发协作：需求拆解、代码解释、Bug 排查、测试生成。
限制
  • 事实风险：回答可能看起来合理，但没有真实依据。
  • 上下文风险：如果背景资料不完整，AI 会按默认经验补全，容易偏离业务。
  • 责任风险：AI 可以辅助判断，但不能替代责任主体。
  • 执行风险：一旦 AI 能调用工具，误操作、越权和数据泄露都需要被控制。
方法
  • 先定义任务类型：是生成内容、查资料、做分析，还是执行动作。
  • 再补充上下文：目标、受众、业务背景、已有资料、不能触碰的边界。
  • 最后设定验收：输出格式、判断标准、需要引用的来源、需要人工确认的部分。
  • 重要任务采用“AI 生成草案，人类核查定稿”的协作方式。
案例

案例：用 AI 辅助做一份竞品分析。

  • 第一步：让 AI 根据产品目标生成竞品分析维度，例如用户群、核心功能、价格、转化路径和差异化。
  • 第二步：提供你已经收集的网页、截图、访谈记录，让 AI 按维度整理信息。
  • 第三步：要求 AI 标出“已知事实”和“推测判断”，避免把猜测当结论。
  • 第四步：由你补充业务判断，形成最终建议和下一步行动。

02 / LLM

大模型 LLM

理解大模型不需要先学数学，先抓住三个关键词：Token、上下文窗口、生成概率。

是什么

大语言模型是用海量文本和代码训练出的语言能力系统。

  • 它把输入内容切成 Token，并在上下文范围内计算接下来更可能出现的内容。
  • 它不是按数据库查询答案，而是基于训练中学到的语言、知识和模式生成回答。
  • 模型越大通常意味着参数、训练数据和计算规模更大，但实际效果还取决于训练质量和产品设计。
能力
  • 语言理解：识别意图、理解问题、提取结构和关系。
  • 语言生成：写作、总结、翻译、改写、风格迁移。
  • 推理辅助：拆解问题、列出假设、给出方案比较，但复杂推理仍需验证。
  • 代码能力：解释代码、生成片段、修复错误、补充测试。
场景
  • 内容和知识：问答、文档总结、课程讲义、知识库助手。
  • 沟通和表达：邮件、汇报、文案、客服回复、翻译。
  • 工作流入口：把自然语言转换成系统动作，例如查询、创建、分类或生成配置。
  • 研发辅助：代码解释、需求转任务、错误定位、测试建议。
限制
  • 知识边界：训练知识可能过期，未接入检索时无法保证最新事实。
  • 上下文边界：一次能看的内容有限，长任务需要分段和总结。
  • 可靠性边界：模型倾向生成流畅答案，但流畅不代表正确。
  • 成本边界：长输入、高频调用和强模型都会带来成本和延迟。
方法
  • 判断模型时，看四件事：准确性、稳定性、速度、成本。
  • 事实问题优先接入 RAG 或搜索；格式问题优先用结构化输出；风格问题再考虑微调。
  • 复杂任务要分阶段：理解任务、生成计划、执行步骤、检查结果。
  • 把模型当“能力层”，而不是完整产品。产品还需要数据、交互、权限、评估和兜底。
案例

案例：用大模型辅助生成课程讲义。

  • 输入课程目标、受众基础、课时长度和已有资料。
  • 先让模型生成大纲，再让它逐章补充讲解、例子和练习题。
  • 要求模型标出需要人工核查的事实、数据和引用。
  • 最终由教师或负责人审查内容准确性和教学节奏。

03 / Model Ecosystem

模型与产品生态

ChatGPT、Claude、Gemini、DeepSeek、Qwen、Llama 不是同一类东西：有的是面向用户的产品，有的是模型家族，有的是开放权重模型生态。

是什么

模型生态可以分成三层：底层模型、开发者 API、面向用户的应用产品。

  • ChatGPT、Claude、Gemini 更常被用户当作“产品入口”使用，背后会接入不同版本的模型和工具能力。
  • GPT、Claude、Gemini、DeepSeek、Qwen、Llama、Mistral 更适合理解为模型家族或模型品牌。
  • 同一个品牌下可能有聊天模型、推理模型、代码模型、多模态模型和小模型，不能只看名字判断能力。
能力
  • 通用问答与写作：适合学习、总结、改写、翻译和资料整理。
  • 推理与分析：适合复杂问题拆解、方案比较和代码理解，但仍需要验证。
  • 多模态：部分模型可以处理图片、音频、视频或文件。
  • 开发者接入：通过 API 把模型嵌入产品、工作流和业务系统。
场景
  • 个人学习：优先从 ChatGPT、Claude、Gemini 这类成熟产品入手，感受能力边界。
  • 业务原型：选择稳定 API，关注结构化输出、工具调用、成本和响应速度。
  • 私有化或开源研究：关注 Llama、Qwen、Mistral 等开放模型生态。
  • 中文场景：关注中文理解、知识覆盖、合规要求和国内可用性。
限制
  • 排行榜不等于你的业务效果，真实效果要看自己的任务样本。
  • 产品能力会频繁更新，同名产品背后的模型也可能变化。
  • 开放权重不等于完全免费，部署、推理、运维和安全都要成本。
  • 涉及隐私、合规、企业数据时，要先看数据使用政策和权限边界。
方法
  • 先按任务分类选模型：写作、检索问答、代码、图像、长文档、自动化执行。
  • 再用一组真实样本比较：准确率、稳定性、速度、成本、可解释性和失败表现。
  • 把产品体验和 API 能力分开评估，不要因为聊天界面好用就默认适合系统集成。
  • 记录每次评估的日期和模型版本，因为 AI 产品变化非常快。
案例

案例：给团队选择日常 AI 工具。

  • 学习和写作：比较 ChatGPT、Claude、Gemini 的长文理解、中文表达和文件处理体验。
  • 开发接入：比较 OpenAI、Anthropic、Google、DeepSeek、Qwen 等 API 的结构化输出、工具调用和价格。
  • 本地部署：评估 Llama、Qwen、Mistral 等开放模型能否满足隐私和成本要求。
  • 最后形成“默认工具 + 备选工具 + 禁止使用场景”的团队说明。

04 / Prompt

Prompt 工程

Prompt 不是咒语，而是把任务目标、角色、约束、步骤、格式和验收标准讲清楚。

是什么

Prompt 是你给 AI 的任务说明，用来定义“要做什么、怎么做、做到什么程度”。

  • System Prompt 更像长期规则，决定角色、边界和行为原则。
  • User Prompt 更像本次任务，说明目标、要求和输出。
  • Prompt Template 是可复用的任务模板，适合团队协作和稳定输出。
能力
  • 明确角色：让 AI 使用合适视角，例如产品顾问、学习教练、代码审查员。
  • 明确任务：让 AI 知道是总结、改写、分析、生成还是审查。
  • 明确约束：限定范围、语气、格式、字数、禁止事项和判断标准。
  • 明确示例：用一两个样例让 AI 模仿结构和质量。
场景
  • 内容场景：写作、润色、标题、脚本、长文摘要。
  • 分析场景：用户反馈归类、方案比较、风险识别、会议纪要。
  • 结构化场景：生成表格、JSON、清单、需求字段、评审模板。
  • 工程场景：描述开发任务、要求代码审查、生成测试用例。
限制
  • Prompt 不能让模型知道它没有的数据。
  • Prompt 不能替代业务判断、法律责任和人工审批。
  • Prompt 不能完全防住恶意输入和越权工具调用。
  • 任务越复杂，越应该拆成多轮，而不是期待一次提示完成。
方法

推荐使用一个稳定框架：角色、任务、背景、约束、步骤、格式、示例、验收标准。

  • 先给结论目标：你希望最终得到什么。
  • 再给背景资料：受众、场景、已有信息、不可改变的约束。
  • 要求 AI 先复述理解或提出缺失问题，避免直接跑偏。
  • 输出后继续追问：哪里不确定、哪些需要验证、还有什么替代方案。
案例

案例：把模糊需求改成可执行 Prompt。

  • 低质量写法：“帮我写个 AI 方案。”
  • 高质量写法：“你是产品顾问，请基于以下业务背景，给管理层写一份 1 页方案，包含目标、用户、核心流程、风险、成本和下一步。”
  • 进一步增强：“如果信息不足，请先列出最多 5 个关键问题，不要直接编造。”

05 / Context

上下文管理

上下文是 AI 当前能看到和参考的信息。Prompt 负责下指令，上下文负责提供判断依据。

是什么

上下文包括本轮输入、历史对话、上传资料、检索结果、代码文件、工具返回值和系统规则。

  • 模型只能基于它看得到的信息做判断。
  • 上下文窗口决定一次任务能放入多少信息。
  • 上下文质量比长度更重要，相关、准确、结构化的信息更有价值。
能力
  • 让 AI 理解业务背景、历史决策和当前约束。
  • 让长任务可以分阶段延续，而不是每次从零开始。
  • 让 AI 在代码库、文档库和项目资料中保持一致判断。
场景
  • 长文档总结、会议纪要整理、项目知识沉淀。
  • AI Coding 中的代码库理解、文件关联和历史修改追踪。
  • Agent 执行多步任务时保存状态、观察结果和下一步计划。
限制
  • 上下文太少会让 AI 猜；上下文太多会引入噪音。
  • 历史信息可能过期，旧结论可能污染新判断。
  • 模型不一定能稳定关注最重要的信息，需要人为组织优先级。
方法
  • 把上下文分层：目标、背景、资料、约束、当前状态、验收标准。
  • 长任务定期让 AI 生成阶段总结，保留决策和未完成 TODO。
  • 代码任务优先提供相关文件和错误日志，而不是整包丢给 AI。
  • 重要资料标注来源、时间和可信度。
案例

案例：让 AI 接手一个需求讨论。

  • 先提供目标、用户、已有方案和约束。
  • 再提供历史决策：哪些方案已否决，原因是什么。
  • 最后让 AI 输出当前理解、缺失问题和下一步建议，避免重复讨论。

  • MCP：Resources
06 / RAG

RAG

RAG 是一种让 AI 先检索资料、再基于资料生成回答的方法。它经常使用知识库，但不等于知识库本身。

是什么

RAG 是“检索增强生成”：先查资料，再让模型基于资料回答。

  • 它不是训练模型，而是把外部知识库接到模型前面。
  • 核心流程是：文档处理、切分、向量化、检索、重排、生成、引用。
  • 它适合解决“模型不知道企业内部资料”或“需要基于指定文档回答”的问题。
能力
  • 知识接入：让模型使用企业制度、产品文档、课程资料和项目材料。
  • 降低幻觉：回答前先找到依据，减少凭空生成。
  • 来源追溯：可以展示引用片段，让用户知道答案从哪里来。
  • 快速更新：更新知识库通常比重新训练模型更轻。
场景
  • 企业内部：制度问答、员工手册、项目文档、流程规范。
  • 客户服务：产品 FAQ、售后政策、故障排查。
  • 教育培训：课程资料问答、知识点解释、题库解析。
  • 专业文档：合同条款、行业报告、技术文档查询。
限制
  • 资料质量差，答案也会差。过期、重复、冲突资料会直接影响结果。
  • 切分不合理会导致语义断裂，模型拿不到完整上下文。
  • 检索不到关键片段时，模型可能仍然硬答。
  • 权限控制复杂：不同用户不一定能看同一批资料。
方法
  • 输入问题后，先把问题转换成适合检索的表示。
  • 从资料库中召回相关片段，再做重排，选出最有用的证据。
  • 把证据和用户问题一起交给模型生成答案。
  • 用真实问题集评估检索命中率、回答准确率、引用可靠性和拒答能力。
案例

案例：内部制度问答助手。

  • 准备资料：收集制度、流程、FAQ，并标记生效日期和适用部门。
  • 建立知识库：按章节切分，生成 Embedding，写入向量数据库。
  • 回答问题：用户提问后先检索相关条款，再生成简洁答案和引用来源。
  • 兜底机制：检索不到或资料冲突时，提示用户联系对应负责人，而不是编造答案。
官方参考
  • OpenAI：Retrieval 与 Vector Stores
  • Anthropic：Citations / Grounding
07 / Knowledge Base

知识库

知识库是被 AI 检索和引用的资料系统。RAG 是使用知识库的一种方法，但知识库还包含资料治理、权限、更新和质量管理。

是什么

知识库是围绕某个业务或主题组织起来的可信资料集合。

  • 可以是制度文档、产品手册、FAQ、项目资料、课程内容或代码文档。
  • 它不只是文件夹，还包括分类、版本、权限、来源和更新机制。
  • 知识库质量决定 RAG 和智能问答的上限。
能力
  • 沉淀组织知识，减少重复问答和信息孤岛。
  • 为 AI 提供可信依据，而不是让模型凭记忆回答。
  • 支持不同角色看到不同权限范围内的资料。
场景
  • 企业制度、客服话术、产品手册、售后政策。
  • 项目文档、研发规范、接口说明、历史决策记录。
  • 培训资料、课程讲义、题库和案例库。
限制
  • 资料过期、冲突或重复，会让 AI 输出不稳定。
  • 没有权限管理，可能导致敏感信息泄露。
  • 没有维护责任人，知识库会快速变成资料堆。
方法
  • 先确定知识范围和使用场景，再整理资料。
  • 为每份资料标注来源、负责人、更新时间和适用范围。
  • 按业务结构切分文档，而不是机械按长度切分。
  • 建立更新机制：新增、过期、冲突、删除都要有人负责。
案例

案例：客服知识库。

  • 把常见问题、产品说明、售后政策和异常处理流程整理成标准资料。
  • 标注每条资料适用的产品版本、地区和生效时间。
  • AI 回答时只引用有效资料，遇到冲突或缺失时转人工。
官方参考
  • OpenAI：Retrieval / Vector stores
  • MCP：Resources
08 / Agent

Agent

Agent 是围绕目标持续行动的 AI 系统。它经常使用工具，但 Agent 不等于工具调用。

是什么

Agent 是带目标、工具、反馈和行动循环的 AI 系统。

  • 普通聊天模型主要回答问题；Agent 会根据目标规划步骤并尝试执行。
  • Agent 的关键组件通常包括：目标、模型、记忆、计划器、执行器、反馈和权限控制。
  • 它不是一个单独按钮，而是一套让 AI 参与流程的系统设计。
能力
  • 计划：把开放任务拆成可执行步骤。
  • 决策：判断下一步该收集信息、执行动作，还是请求人工确认。
  • 观察反馈：根据工具返回结果判断下一步。
  • 持续迭代：在目标未完成时继续补充信息、修正计划或请求人工确认。
场景
  • 信息型任务：调研、资料整理、报告草稿、竞品分析。
  • 流程型任务：工单分派、客服辅助、报表生成、日程整理。
  • 工程型任务：读代码、改文件、运行测试、汇报差异。
  • 跨系统任务：从一个系统读数据，到另一个系统创建记录或发送通知。
限制
  • 开放目标容易失控，Agent 可能走向不必要或错误的步骤。
  • 工具权限越大，误操作和数据泄露风险越高。
  • 长期任务会遇到上下文遗忘、状态混乱和错误累积。
  • 外部内容可能包含恶意指令，诱导 Agent 越权执行。
方法
  • 定义任务边界：Agent 负责什么，不负责什么。
  • 定义状态管理：需要记住哪些信息，哪些信息不能长期保存。
  • 设置中间检查点：计划确认、执行确认、结果确认。
  • 记录过程日志：每次调用工具、读取资料、做出判断都要可追踪。
  • 设计终止条件：什么时候算完成，什么时候应该停止并交给人。
案例

案例：竞品资料整理 Agent。

  • 目标：围绕指定产品收集竞品公开信息，并生成对比表。
  • 工具：搜索网页、读取网页内容、整理表格，不允许自动发布结论。
  • 过程：先生成调研计划，人确认后再执行搜索和整理。
  • 结果：输出事实表、信息来源、缺失项和初步观察；最终判断由人完成。

09 / Agent Ecosystem

Agent 工具生态

Agent 工具不是越“自动”越好。Claude Code、Codex、OpenHands、Browser Use、OpenClaw、Hermes 等名称，真正要比较的是它们能做什么、在哪里运行、拥有什么权限、过程是否可控；其中 OpenClaw、Hermes 这类名称需要先确认具体项目来源。

是什么

Agent 工具生态是一组让 AI 从“聊天回答”进入“执行任务”的产品、框架和运行环境。

  • 代码型 Agent：偏向读代码、改文件、跑测试、生成 PR，例如 Claude Code、Codex、OpenHands。
  • 浏览器型 Agent：偏向网页浏览、表单操作、页面自动化，例如 Browser Use。
  • 个人自动化 Agent：偏向消息、日历、文件、本地工具和长期任务，例如 OpenClaw、Hermes 这类需要二次核验来源的生态项目。
  • 框架型工具：偏向开发者搭建自己的 Agent，例如 LangGraph、AutoGen、CrewAI、LangChain、LlamaIndex。
能力
  • 任务执行：把目标拆成步骤，调用文件、浏览器、终端、API 或企业系统。
  • 上下文读取：读取代码库、网页、文档、历史对话和工具返回结果。
  • 长期协作：在多轮任务中保留状态、生成计划、跟进失败和汇报进度。
  • 多 Agent 协作：让不同角色分工，例如实现、审查、测试和总结。
场景
  • 研发提效：让 Agent 修 Bug、改页面、跑检查、写变更说明。
  • 运营自动化：整理网页资料、处理表格、生成报告、跨系统录入信息。
  • 个人助手：通过消息入口处理日程、提醒、文件和日常任务。
  • 企业流程：把 Agent 接入审批、工单、知识库和 DevOps 流程。
限制
  • 生态变化快，工具名称、能力、价格和可用入口都会变。
  • 同名项目可能不止一个，例如 Hermes、OpenClaw 相关信息需要回到项目主页或仓库确认。
  • Agent 能操作真实系统时，权限、日志、沙箱和人工确认比炫酷演示更重要。
  • 开源项目不等于安全可靠，需要检查维护状态、许可证、社区、依赖和插件来源。
方法
  • 先按使用场景分类：代码、浏览器、办公自动化、个人助理、企业流程。
  • 再看运行位置：云端、本地、IDE、终端、浏览器、企业平台。
  • 最后看控制能力：权限分级、审批节点、运行日志、回滚机制、成本监控。
  • 把 OpenClaw、Hermes 这类快速变化项目标为“生态观察”，不要直接当成稳定标准答案。
案例

案例：给团队选一个代码 Agent。

  • 如果团队想在本地终端协作，优先比较 Claude Code、Codex CLI、OpenCode 这类工具的上下文读取和权限控制。
  • 如果团队想处理 GitHub issue 或 PR，比较云端沙箱、分支隔离、测试运行和代码审查能力。
  • 如果需要自托管和可扩展，关注 OpenHands、LangGraph、AutoGen 等框架或平台。
  • 正式使用前，用 5 个真实任务试跑，并记录失败类型和人工接管点。

10 / Tool Use

工具调用

工具调用是让 AI 使用外部能力的机制。它可以被 Agent 使用，也可以在普通 AI 应用中单独使用。

是什么

工具调用是把 AI 的自然语言判断转成系统可执行动作。

  • 常见形式包括 Function Calling、插件、MCP、API 调用和本地工具调用。
  • 模型负责判断“要调用什么工具、传什么参数”。
  • 系统负责真正执行工具、校验参数、返回结果和控制权限。
能力
  • 查资料：搜索网页、查询数据库、读取知识库。
  • 读写文件：读取文档、修改代码、生成报告。
  • 操作系统：创建日程、发送消息、生成图片、运行脚本。
  • 连接业务：查询订单、创建工单、更新状态、触发审批。
场景
  • 客服助手查询订单和售后政策。
  • 办公助手创建日程、整理邮件和生成文档。
  • 代码助手读取文件、执行测试和生成变更摘要。
  • 数据助手查询报表、运行分析脚本和生成图表。
限制
  • 工具参数可能被模型填错，需要严格校验。
  • 工具返回结果可能失败、超时或不完整。
  • 高权限工具会带来误删、误发、越权和泄露风险。
  • 外部内容可能诱导模型调用不该调用的工具。
方法
  • 工具按风险分级：只读、可写、高风险、不可逆。
  • 高风险动作必须有人确认，例如删除、付款、发送、部署。
  • 每次工具调用记录参数、结果、时间和触发原因。
  • 工具结果返回后，让模型基于结果继续判断，而不是假设成功。
案例

案例：客服助手查询订单。

  • 用户问“我的退款到哪了”。
  • AI 判断需要调用订单查询工具，并提取订单号或请求用户补充。
  • 系统校验权限后查询订单状态，把结果返回给 AI。
  • AI 基于真实状态回复用户；如果要发起退款操作，则必须人工确认。

  • MCP：Tools / Resources / Prompts
11 / MCP

MCP

MCP 是连接 AI 应用和外部系统的开放协议。它让 AI 更标准地访问数据源、工具和可复用提示词。

是什么

MCP 的全称是 Model Context Protocol，可以理解为 AI 应用连接外部能力的标准接口。

  • 它不是模型，也不是某个具体工具，而是一套连接协议。
  • 它让 AI 客户端可以发现和调用 MCP Server 暴露的能力。
  • 官方文档常把它类比成 AI 应用的 USB-C：一次标准化连接，接入多种外部系统。
能力
  • Tools：让 AI 执行动作，例如搜索、查询、创建、更新。
  • Resources：让 AI 读取上下文资料，例如文件、数据库、日历、文档。
  • Prompts：提供可复用的提示词模板，引导模型正确使用工具和资源。
  • Servers：把某个系统的能力封装出来，供 AI 客户端连接。
场景
  • 让 AI 访问本地文件、代码仓库、数据库、设计工具和企业系统。
  • 为多个 AI 客户端复用同一套工具接入能力。
  • 把内部系统包装成标准能力，减少每个 AI 应用重复开发。
限制
  • MCP 解决连接标准问题，不自动解决权限、安全和业务正确性。
  • Server 暴露能力越强，越需要访问控制、日志和人工确认。
  • 不同客户端对 MCP 能力的支持程度可能不同。
方法
  • 先判断要暴露的是资料、动作，还是提示模板。
  • Tools 保持单一职责，输入输出使用清晰 schema。
  • Resources 标注 URI、MIME 类型、描述和权限。
  • 高风险工具必须加确认和审计，不要默认让模型自由执行。
案例

案例：给 AI 接入项目知识库和任务系统。

  • MCP Server 暴露项目文档作为 Resources。
  • 暴露查询任务、创建任务、更新状态等 Tools。
  • 提供“整理本周项目进展”的 Prompt 模板。
  • AI 客户端读取资料、查询任务，生成周报草稿；涉及更新任务状态时需要人工确认。
官方参考

  • MCP：Server concepts
  • MCP：Architecture overview
12 / Skill

Skill

Skill 是给 AI 或 Agent 使用的可复用能力包，用来沉淀某类任务的流程、规则、边界和输出标准。

是什么

Skill 可以理解为“任务方法说明书”：遇到某类任务时，AI 应该按这套方法执行。

  • 它通常包含适用场景、操作步骤、注意事项、可用脚本、模板和输出格式。
  • 它不是模型能力本身，而是把经验和流程封装成可复用的上下文。
  • Skill 可以让不同任务保持一致质量，减少每次重新写 Prompt 的成本。
能力
  • 流程复用：把高频任务固化成标准步骤。
  • 质量控制：明确输出标准、检查项和常见错误。
  • 工具协同：告诉 Agent 什么时候读文件、运行脚本、调用工具。
  • 知识沉淀：把团队经验变成 AI 可以遵循的操作指南。
场景
  • 代码审查、技术文档生成、PR 总结、Bug 排查。
  • 市场研究、SEO 审计、产品需求分析、竞品分析。
  • 设计评审、图表生成、飞书/Notion/文档工作流。
限制
  • Skill 写得太泛，会退化成普通提示词。
  • Skill 写得太死，会限制 AI 根据实际上下文调整。
  • 过期 Skill 会把旧流程、旧工具和旧判断带入新任务。
方法
  • 先定义触发条件：什么任务应该用这个 Skill。
  • 再定义工作流：先读什么、再判断什么、最后产出什么。
  • 补充边界：哪些事情不能自动做，哪些动作必须确认。
  • 沉淀模板和检查清单，让输出更稳定。
案例

案例：代码审查 Skill。

  • 触发：用户要求 review 代码或检查 PR。
  • 流程：先看 diff，再找行为风险、边界遗漏、安全问题和测试缺口。
  • 输出：优先列问题，给文件位置、严重级别和建议。
  • 边界：不顺手重构，不修改无关文件，不把格式问题当主要发现。

13 / Agent Harness

Agent Harness

这里的 Harness 是一种工程归纳，不是统一官方标准名。它可以理解为 Agent 的运行支架：不负责“变聪明”，而是负责让 Agent 按规则运行、留下过程、失败可恢复、结果可评估。

是什么

Agent Harness 是本文用来描述 Agent 运行支撑层的概念，包括任务协议、工具适配、状态管理、权限控制、评估和日志。

  • 它是一种工程方法和归纳口径，不特指某个模型，也不是所有官方文档都会使用的统一术语。
  • 它也不同于 Harness 这家公司或平台；后者是具体 DevOps/AI 平台，前者是通用概念。
  • 一个好的 harness 会把“模型想做什么”和“系统允许做什么”分开处理。
能力
  • 约束输入输出：让任务有固定输入、输出、状态和验收标准。
  • 管理工具：封装工具调用、参数校验、权限审批和结果回传。
  • 记录过程：保存计划、工具调用、错误、成本、耗时和人工接管点。
  • 恢复失败：失败后可以重试、降级、回滚或交给人处理。
场景
  • AI Coding：让 Agent 在隔离环境中改代码、跑测试、产出变更摘要。
  • 企业自动化：让 Agent 按流程调用系统，关键动作必须审批。
  • RAG 问答：记录检索内容、引用来源和回答质量。
  • 多 Agent 协作：让不同 Agent 的交接、状态和输出保持一致。
限制
  • Harness 不能让错误模型变成可靠专家，只能提高过程可控性。
  • 过度复杂的 harness 会让系统难维护，简单任务不需要重型框架。
  • 如果验收标准本身模糊，再完整的日志也难判断结果是否正确。
  • 监控和日志可能包含敏感数据，需要访问控制和保留策略。
方法
  • 写清任务协议：目标、输入、可用工具、禁止动作、输出格式、完成标准。
  • 工具按风险分层：只读、可写、高风险、不可逆，并设置不同确认方式。
  • 建立运行记录：每一步为什么做、调用了什么、返回了什么、花了多少成本。
  • 设计失败路径：重试几次、何时降级、何时停止、何时转人工。
案例

案例：代码修复 Agent 的 harness。

  • 输入：GitHub issue、相关日志、仓库路径和验收标准。
  • 权限：允许读文件、改工作区、运行测试；禁止推送远端和修改生产配置。
  • 过程：先输出计划，人确认后修改；每次测试失败都记录原因。
  • 输出：改动文件、验证命令、失败残留、风险点和需要人工 Review 的位置。

14 / AI Coding

AI Coding 工作流

AI Coding 把人的工作重心从“逐行写代码”推向“描述任务、提供上下文、审查结果和验证行为”。

是什么

AI Coding 是让 AI 参与软件开发的工作方式，而不只是“自动写代码”。

  • 它覆盖需求理解、代码阅读、方案设计、文件修改、调试、测试和代码审查。
  • 人的角色会从逐行实现，更多转向描述目标、提供上下文、控制风险和验收结果。
  • 越是已有代码库，越需要让 AI 先理解项目结构和局部上下文。
能力
  • 理解：解释函数、组件、调用链和业务流程。
  • 生成：创建脚本、页面、接口、测试和文档。
  • 修改：修 Bug、补交互、调整样式、接入数据。
  • 验证：运行检查、解释报错、生成测试用例、做初步 Review。
场景
  • 低风险高收益：原型页面、内部工具、数据处理脚本、文档生成。
  • 明确边界任务：修一个 Bug、改一个组件、补一个测试、解释一段代码。
  • 协作学习：帮助非技术同学理解开发流程和技术约束。
  • 不适合一上来全自动处理复杂核心系统、支付、安全和权限逻辑。
限制
  • 业务误解：代码能运行，但不符合真实业务规则。
  • 上下文不足：只看局部文件，漏掉其他调用方。
  • 测试缺失：AI 可能没有覆盖关键边界条件。
  • 风格不一致：引入不符合项目习惯的写法或过度抽象。
  • 高风险改动：数据库、权限、安全、支付和部署需要更严格审查。
方法
  • 任务描述：说明用户场景、期望行为、影响范围和验收标准。
  • 上下文准备：让 AI 先读相关文件、现有模式和错误日志。
  • 小步修改：一次只改清晰边界内的问题，避免大范围顺手重构。
  • 结果验收：要求 AI 汇报改了什么、为什么改、怎么验证、还有什么风险。
  • 人类审查：关键代码必须 Review，不能只看 AI 的自我总结。
案例

案例：让 AI 辅助修改一个筛选功能。

  • 描述需求：用户希望按标签筛选内容，筛选后列表和空状态要同步变化。
  • 读取上下文：让 AI 找到列表组件、数据结构和现有筛选逻辑。
  • 执行改动：只修改筛选相关文件，不做无关样式和架构调整。
  • 验证结果：运行检查，手动点击标签，确认空状态、移动端布局和数据数量都正确。

15 / Agentic Engineering

Agentic Engineering

当 AI 能调用工具、修改文件、执行流程时，工程重点会从“会不会生成”转向“是否可控、可验证、可追踪”。

是什么

Agentic Engineering 是面向 AI Agent 的工程方法：让 AI 能参与任务，同时保持可控、可测、可追踪。

  • 它不是单纯写 Prompt，也不是让 AI 完全自动接管工作。
  • 它关注任务如何拆解、工具如何接入、权限如何限制、结果如何评估、失败如何恢复。
  • 它把 AI 从“单次回答”放进真实工程流程里。
能力
  • 任务编排：把复杂目标拆成可执行、可验证的阶段。
  • 工具接入：让 Agent 安全使用文件、浏览器、数据库、代码仓库和业务 API。
  • 过程控制：记录计划、动作、工具调用、输出和错误。
  • 质量闭环：用测试、评估、人工 Review 和反馈数据持续改进。
场景
  • 研发：需求拆解、代码修改、测试运行、PR 总结、CI 问题排查。
  • 知识库：自动发现过期文档、归类新资料、补充术语表。
  • 运营：跨系统拉取数据、生成日报、整理异常并提醒负责人。
  • 产品：把用户反馈聚类，生成问题清单和需求机会。
限制
  • 开放任务难评估：如果目标不清楚，就很难判断 Agent 做得好不好。
  • 执行链路长：中间一步错，后面可能持续放大错误。
  • 权限风险高：读写文件、调用接口、部署代码都必须分级控制。
  • 可观测性要求高：看不到过程，就很难定位问题和追责。
方法
  • 定义任务协议：输入是什么、输出是什么、完成标准是什么。
  • 设计权限模型：哪些工具只读，哪些工具可写，哪些动作必须确认。
  • 建立评估体系：准确率、完成率、人工接管率、失败原因、成本和耗时。
  • 保留回滚机制：文件改动、数据变更和业务动作都要能追踪和恢复。
  • 引入人工节点：计划确认、关键动作确认、最终结果验收。
案例

案例：可控的 AI 工程工作流。

  • 阶段 1：Agent 阅读需求和相关代码，输出实施计划、影响范围和风险点。
  • 阶段 2：人确认计划后，Agent 执行小步代码修改，并保留变更摘要。
  • 阶段 3：Agent 运行测试和检查，失败时先分析原因，不直接扩大修改范围。
  • 阶段 4：最终输出改动文件、验证结果、未覆盖风险和需要人工确认的点。
